- 비구현 범위: 방어구·악세서리, +11~+100, 파괴·복원, 전쟁, 관전, 베팅, 직원, 저장 마이그레이션

## 1. 문서 목적

이 명세는 확정된 프로젝트 코어를 첫 번째 외부 플레이 검증이 가능한 소프트웨어 범위로 변환한다. 목표는 기능 수를 늘리는 것이 아니라 다음 한 가지 가설을 반증 가능하게 만드는 것이다.

> 플레이어가 직접 만든 철검에 애착을 느끼고, 그 철검이 다른 사람의 손에서 어떤 결과를 만들었는지 다시 확인하려 하는가?

이 명세의 성공은 프로덕션 승인과 동일하지 않다. 외부 플레이어 행동 증거가 수집되기 전까지 장비 종류, 이벤트 수, 세계 시뮬레이션 규모와 최종 강화 상한은 확정하지 않는다.

### 시장 포지션

가장 가까운 경쟁 범주는 대장간 경영·제작 시뮬레이션이다. 그러나 다수 경쟁작은 작업장 관리, 세부 공정, 직원·시설 또는 생산 수량을 전면에 둔다.

> 권장 차별화 문장: 장비를 대량 생산하는 게임이 아니라, 장비의 출생·성장·소유·사건 기록을 제작하는 대장장이 게임.

## 5. 검토 대안과 선택

### A. 기존 `game_flow_screen.gd`에 모든 기능 추가

- 장점: 파일 수와 초기 연결 작업이 적다.
- 단점: 이미 제작·강화·보관·자동 단조를 조정하는 화면이 고객·날짜·세계 사건까지 소유하게 된다.
- 판정: `REJECT`. 테스트와 변경 격리가 급격히 악화된다.

### B. 도메인 모델 + 얇은 흐름 컨트롤러

- 장점: 피로도, 고객 평가, 세계 장비 기록과 UI를 독립 테스트할 수 있다.
- 단점: 초기 파일 수가 늘어난다.
- 판정: `ADOPT`. 현재 프로젝트 구조와 PoC 규모에 가장 적합하다.

### C. 범용 이벤트 버스·퀘스트 프레임워크 선구축

- 장점: 장기 확장성이 높다.
- 단점: 철검 1종 PoC에 불필요한 추상화와 디버깅 비용을 만든다.
- 판정: `DEFER`. 두 번째 고객과 두 번째 세계 사건을 구현한 뒤 필요성을 재평가한다.

### 7.1 완성도 명칭

정밀 결과의 `완벽`과 영구 등급의 `완벽한`이 겹치는 문제를 제거한다.

| ID | 표시명 | 공격 배율 | 가치 배율 | 세계 평가 점수 |
|---|---|---:|---:|---:|
| `ROUGH` | 미숙한 | 0.98 | 0.92 | 0 |
| `STANDARD` | 평범한 | 1.00 | 1.00 | 5 |
| `REFINED` | 정교한 | 1.02 | 1.05 | 10 |
| `EXCELLENT` | 명품 | 1.04 | 1.10 | 15 |
| `MASTERPIECE` | 걸작 | 1.06 | 1.15 | 20 |

공격력 영향은 작게 유지하고 가치·평가·표현에서 차이를 더 크게 만든다. 낮은 등급도 +10, 수식어, 납품과 모든 PoC 결과에 접근할 수 있다.

### 책임 경계

- `GameFlowScreen`: 화면 전환과 UI 신호 연결만 담당한다.

- `WorkshopCalendar`: 날짜와 피로도만 소유한다.

- `EquipmentWorldRegistry`: 장비 영구 기록과 상태 전이를 관리한다.
- `WorldActivityResolver`: 장비 결과 밴드와 설명 가능한 기여 항목을 계산한다.

## 10. 제작과 영구 완성도

### 10.1 정밀 결과와 완성도 분리

- 정밀 마감 입력은 `STANDARD / GOOD / PERFECT` 결과를 만든다.
- 정밀 결과는 완성도 확률 분포를 바꾼다.
- PoC에서는 숙련도 프로필을 고정해 정밀 입력과 완성도 체감을 먼저 검증한다.
- 테스트는 난수 오버라이드로 결과를 재현한다.

### 10.2 초기 분포

| 정밀 결과 | 미숙한 | 평범한 | 정교한 | 명품 | 걸작 |
|---|---:|---:|---:|---:|---:|
| STANDARD | 20% | 55% | 22% | 3% | 0% |
| GOOD | 5% | 40% | 43% | 11% | 1% |
| PERFECT | 0% | 15% | 50% | 30% | 5% |

이 값은 PoC 기준선이다. 정확한 확률은 `CHANGEABLE`이며 한 번에 한 변수군만 조정한다.

## 11. 강화와 멈춤 판단

### 11.1 일반 강화

- 버튼 한 번당 강화 판정 한 번
- +1~+9는 빠른 결과 표시
- 시도마다 골드와 일반 강화 피로도 1 소비
- 결과 확인 전에 다음 시도를 자동 실행하지 않음

### 11.2 특수 강화

- +10 도달 시 보조재료, 촉매, 정밀 판정, 수식어 후보 선택
- 특수 강화 피로도 3 소비
- 화염석은 `flaming`, 숫돌은 `sharp` 계열을 우선 생성

### 11.3 PoC 선택 구조

- +5: 고객 의뢰의 최소 조건을 충족한다.
- +6~+9: 추가 가치와 공격력, 피로도·비용의 교환이다.
- +10: 첫 수식어와 압도적 결과 가능성을 얻는 선택적 목표다.
- 납품 기한과 피로도 때문에 +10이 항상 최적이 되어서는 안 된다.

## 12. 피로도와 날짜

### 12.1 기준값

| 행동 | 피로도 |
|---|---:|
| 철검 제작 시작 | 3 |
| 일반 강화 1회 | 1 |
| +10 특수 강화 | 3 |
| 중요 장비 복원 | 5, PoC 미사용 |

- 기본 피로도: 20
- 이월 비율: 남은 피로도의 50%, 소수점 버림
- 날짜는 `하루 마치기` 입력으로만 진행
- 작업 예약 없음
- 자동 날짜 진행 없음

### 12.2 다음 날 계산

```text
다음 날 피로도 = 기본 피로도 + floor(현재 남은 피로도 × 0.5)
```

무작업 날짜를 반복하면 기본 피로도의 두 배에 수렴한다. 별도 상한은 두지 않지만 PoC에서 무작업 날짜 비율과 이벤트 직전 몰아 제작 비율을 측정한다.

### 12.3 실패 처리

- 피로도 부족 시 작업을 시작하지 않는다.
- 골드·재료·피로도 중 하나라도 부족하면 어떤 자원도 차감하지 않는다.
- 강화 성공 확률과 제작 완성도는 낮은 피로도 때문에 불리해지지 않는다.

## 13. 고객 납품

### 13.1 납품 가능 조건

- 무기 ID가 `iron_sword`
- 강화 단계 +5 이상
- 의뢰 기한 이내
- 파괴되지 않은 보관 장비

### 13.2 납품 선택 정보

기본 화면에는 다음만 표시한다.

- 완성도
- 강화 단계
- 대표 수식어 1개
- 고객 요구 충족 여부
- +10 추가 도전 시 얻는 이점
- 남은 날짜와 피로도

상세 수치와 전체 이력은 펼침 영역으로 분리한다.

### 13.3 납품 결과

- 장비는 보관함에서 제거한다.
- 고객에게 즉시 소유권을 이전한다.
- 기본 대금과 즉시 명성을 지급한다.
- 세계 장비 기록을 생성한다.
- 결과 보고는 최소 다음 날에 준비한다.

## 14. 세계 결과 인과성

### 14.1 결정 점수

| 항목 | 점수 |
|---|---:|
| +5 요구 조건 충족 | 25 |
| +10 추가 목표 충족 | 15 |
| 선호 수식어 보유 | 25 |
| 완성도 등급 | 0 / 5 / 10 / 15 / 20 |
| 공격력 기준 충족 | 20 |

### 14.2 결과 밴드

| 점수 | 결과 |
|---:|---|
| 0~39 | 패배 |
| 40~69 | 승리 |
| 70 이상 | 압도적 승리 |

난수는 결과 밴드를 바꾸지 않는다. 동일 밴드 안에서 문장 변형, 골드 보너스 ±10%, 부상 여부 같은 세부 강도만 변화시킨다.

### 14.3 보고서 구성

1. 어떤 선택이 효과가 있었는지
2. 어떤 조건이 부족했는지
3. 어떤 사건이 발생했는지
4. 장비 이력에 무엇이 추가됐는지
5. 명성·관계·다음 의뢰가 어떻게 바뀌었는지

## 15. 세계 잔존과 활동 집합

- 납품된 모든 장비의 기록은 삭제하지 않는다.
- 새 사건을 생성할 수 있는 `ACTIVE / EVENT_ELIGIBLE` 장비는 제한한다.
- PoC 활동 장비 상한은 6개다.
- 활동 상한을 넘는 장비는 `ARCHIVED` 상태로 전환하지만 이력은 유지한다.
- PoC에서는 검투사 카일의 철검 1개만 활성화한다.

## 16. 재방문

- 결과 보고를 플레이어가 열어 본 뒤 재방문 플래그가 활성화된다.
- 다음 날 검투사 카일이 이전 장비의 결과를 직접 언급한다.
- 재방문 의뢰는 이전 결과의 부족 조건을 반영한다.
- 첫 PoC에서는 새 장비 제작 시작 버튼까지만 제공하고 두 번째 전체 생애는 구현하지 않는다.

## 18. 행동 로그와 플레이테스트

### 18.1 이벤트

| 이벤트 | 핵심 필드 |
|---|---|
| `poc_contract_viewed` | day, fatigue, deadline |
| `poc_forging_completed` | precision_result, craftsmanship_grade, tap_count |
| `poc_enhancement_attempted` | from_level, target_level, outcome, fatigue_after |
| `poc_enhancement_stopped` | stop_level, remaining_days, remaining_fatigue |
| `poc_equipment_discarded` | grade, level, reason |
| `poc_equipment_delivered` | grade, level, affixes, fit_score |
| `poc_day_ended` | unused_fatigue, carryover, pending_reports |
| `poc_world_report_opened` | outcome_band, contribution_ids |
| `poc_follow_up_started` | prior_outcome, prior_equipment_id |

### 18.2 플레이테스트 성공 신호

- 플레이어가 철검을 완성도·강화·수식어 중 둘 이상으로 기억한다.
- +5에서 납품할지 +10을 노릴지 실제로 멈춰 고민한다.
- 결과 보고를 자발적으로 연다.
- 결과 원인 중 둘 이상을 자기 말로 설명한다.
- 재방문을 본 뒤 새 장비 제작을 시작한다.
- 낮은 완성도 장비의 즉시 폐기율이 50% 미만이다.
- 날짜 넘기기만 반복하는 세션이 20% 미만이다.

### 18.3 중단 신호

- 결과 보고보다 골드 숫자만 기억한다.
- 완성도 등급이 낮으면 대부분 즉시 재제작한다.
- +10이 항상 명백한 정답이라 멈춤 고민이 없다.
- 세계 결과가 제작 선택과 무관하다고 설명한다.
- 피로도 때문에 플레이가 끊기고 날짜만 넘긴다.

## 19. 오류 처리

| 오류 | 처리 |
|---|---|
| 피로도 부족 | 부족량 표시, 무차감, 날짜 자동 진행 금지 |

| UI 화면 손실 | 컨트롤러 스냅샷으로 현재 상태 재구성 |

## 21. 완료 기준

통합 PoC 구현 완료는 다음을 모두 만족할 때만 선언한다.

1. 철검 제작부터 검투사 재방문까지 한 세션에서 완주된다.
2. +5 납품과 +10 도전이 모두 유효한 선택이다.
3. 골드·재료·피로도 차감이 원자적으로 처리된다.
4. 결과 밴드는 제작 선택으로 결정되고 난수는 세부 강도만 바꾼다.
5. 장비 기록이 납품 뒤에도 유지된다.
6. 모든 신규 단위·통합 테스트와 기존 회귀 테스트가 통과한다.
7. Godot import·main scene smoke가 통과한다.
8. Android·접근성·외부 플레이는 별도 상태로 기록되며 미실행을 PASS로 표시하지 않는다.

## 22. KEEP / AMPLIFY / CHANGE / DEFER

### KEEP

- 한 명의 대장장이
- 직접 제작과 빠른 터치 피드백
- 버튼 입력당 강화 1회
- +10 특수 강화와 수식어 선택
- 판매 장비의 세계 이력
- 수동 날짜 진행과 50% 피로도 이월

### AMPLIFY

- +5 납품과 +10 욕심의 멈춤 판단
- 결과 보고의 기여·부족 조건 설명
- 같은 고객 재방문
- 낮은 등급 장비의 다른 역사 가능성
- 장비 상세 이력과 별칭

### CHANGE

- 정밀 결과와 영구 완성도 용어 중복
- 모든 판매 장비가 계속 사건을 만드는 구조
- 자동 단조가 위험 구간까지 무중단 진행하는 구조
- 결과 밴드를 난수가 직접 뒤집는 구조
- `game_flow_screen.gd`가 모든 도메인을 직접 소유하는 구조

### DEFER

- 최종 강화 상한 확정
- 방어구·악세서리 확장
- 전쟁·토너먼트 관전·베팅
- 대표작·복원
- 다수 고객·세력·시장
- 범용 이벤트 버스와 퀘스트 프레임워크
