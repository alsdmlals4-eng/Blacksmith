# 위험·가격 곡선 시뮬레이션 범위

## 상태

- 범위·입력·지표 계약: READY
- 시뮬레이터 구현: NOT_STARTED
- 기준선 실행: NOT_RUN
- 수치 조정: NOT_STARTED
- 실제 플레이·Android 검증: NOT_RUN

이 문서는 POC v0.6.4의 강화 실패 정책 정합성 완료 뒤 수행할 위험·가격 곡선 검증의 책임 원본이다. 게임 수치 자체는 각 JSON이 소유하며, 이 문서는 무엇을 어떤 증거로 비교하고 언제 변경할지를 정의한다.

## 결정 질문

1. +10·+30·+50·+70·+100 목표까지 필요한 시도 횟수·골드·재료의 분포가 목표 세션과 장기 성장에 맞는가?
2. +11 단계 하락과 +30 파괴가 긴장을 만들면서도 회복 불가능한 손실 피로를 만들지 않는가?
3. 균형·안정·폭주 단조가 한 가지 무료 최적해로 수렴하지 않고 서로 다른 위험·비용 선택을 제공하는가?
4. 제작 품질·피버 결과 보너스가 강화 복리와 결합해 수동 제작 또는 자동 단조 한쪽을 압도하지 않는가?
5. 자동 단조의 목표·반복·재료 fallback·보관함·파괴·골드 부족 중지 정책이 유한하며 장기 경제를 무너뜨리지 않는가?
6. 방문 검투사 판매를 연결하기 전에 판매가가 강화 비용·손실·재료 소비를 회수할 수 있는 합리적 범위인가?

## 입력 책임 원본

### 수치·데이터

- `data/crafting/enhancement_balance.json`: 성공률·실패 보정·성장·비용·가격·단조 방식·하락·파괴
- `data/crafting/enhancement_milestones.json`: 일반/특수 강화 분류와 +10 단위 수식어 이정표
- `data/crafting/forging_balance.json`: 제작 품질·피버·합산 제작 보너스
- `data/crafting/materials.json`: 보조재료·촉매 가격과 효과
- `data/crafting/affixes.json`: 수식어 성장 효과

### 실제 판정 순서·상태 전이

- `scripts/enhancement/enhancement_session.gd`: 강화 판정·보정·하락·파괴·판매가
- `scripts/economy/workshop_resources.gd`: 골드·재료 거래와 시작 실패 복구
- `scripts/ui/enhancement_screen.gd`: 수동·자동 강화 진입과 중지 조건
- `scripts/ui/game_flow_screen.gd`: 제작 결과 전달·자동 반복·보관
- 관련 단위·통합 테스트: `tests/unit/`, `tests/integration/`

문서 예시 수치, 화면 표시 문구, 과거 Changelog, 닫힌 PR 브랜치는 시뮬레이션 입력값으로 사용하지 않는다.

## 기준선 시나리오

- 시작 무기: 보통·좋음·완벽, 피버 미적용·적용
- 목표 단계: +10·+30·+50·+70·+100
- 단조 방식: 균형·안정·폭주
- 특수 강화 재료: 보조재료별·촉매 없음·각 촉매
- 실행 정책: 수동 단일 무기, 목표 자동 단조, 보관함 반복
- 자원 조건: 충분한 자원, 골드 병목, 보조재료 병목, 촉매 병목
- 파괴 처리: 반복 OFF 종료, 반복 ON 새 철검 재시작

폭주는 특수 강화와 +9 끝자리에서 사용하지 못하는 실제 계약을 그대로 재현한다. 자동 fallback은 지정 재료가 없을 때 빈 슬롯으로 동일 거래 경로를 사용하는 현재 정책을 따른다.

## 측정 지표

평균만으로 판정하지 않는다.

- 목표 도달률
- 시도 횟수의 평균·중앙값·P10·P90·P95
- 소모 골드·보조재료·촉매의 평균·중앙값·상위 분위
- 성공·유지·단계 하락·파괴 횟수와 구간별 빈도
- 실패 보정 누적 분포와 성공 전 최대 누적치
- 파괴 후 원래 목표로 복귀하는 데 필요한 추가 시도·골드·재료
- 최종 공격력·판매가·누적 강화비·예상 손익 분포
- 단조 방식·재료 선택별 비용 대비 목표 도달 시간과 분산
- 자동 단조 중지 사유 비율: 목표 도달·골드 부족·보관함 가득 참·수동 중지·파괴 종료

## 실험 계약

1. 입력 JSON과 런타임 판정 순서를 고정한 기준선부터 재현한다.
2. 고정 시드 반례와 충분한 반복 표본을 함께 사용한다.
3. 성공률·비용·실패 보정·하락·파괴·피버·판매가 중 한 실험에서는 하나의 변수군만 바꾼다.
4. 변경 전에 가설·예상 방향·가드레일·중단 조건을 기록한다.
5. 기준선과 후보안을 동일 시드·동일 시나리오로 비교한다.
6. 시뮬레이터 결과가 실제 `EnhancementSession`의 결정적 테스트와 불일치하면 수치 조정 전에 시뮬레이터를 수정한다.
7. 현재 JSON과 게임 수치는 사용자 승인과 결과 검토 전 변경하지 않는다.

## 판정

- `KEEP`: 기준선이 가드레일 안에 있고 변경 근거가 없음
- `TUNE`: 한정된 변수 조정으로 명확한 개선이 예상됨
- `REJECT`: 후보안이 핵심 재미·경제·위험 선택을 악화함
- `TEST_IN_PLAY`: 분포상 차단 문제는 없지만 손실 피로·입력 재미·보상 체감은 실제 플레이가 필요함

시뮬레이션은 실제 손맛·피로·UI 이해·Android 성능을 PASS 처리할 수 없다.

## 산출물 계약

실행 단계에서 다음을 추가한다.

- 재현 가능한 시뮬레이터와 실행 방법
- 입력 파일 해시 또는 commit SHA
- 시드·반복 수·시나리오 설정
- 기계 판독 결과 JSON 또는 CSV
- `docs/BALANCE_SIMULATION_REPORT.md`: 기준선·후보안·분포·판정·미검증 요약
- 수치 변경이 있으면 JSON·소비자·테스트·Game Bible·Active Context·Roadmap·Gates·Decision Log·Changelog 동기화

임시 대용량 결과는 GitHub Actions artifact로 보관하고 정본 저장소에는 요약과 재현 방법만 남긴다.

## 완료 기준

- 시뮬레이터가 실제 판정 순서와 고정 반례를 재현한다.
- 모든 기준선 시나리오에 평균·중앙값·상위 분위·실패·하락·파괴·자원 병목 결과가 있다.
- 각 후보안은 한 변수군만 변경하고 동일 입력으로 비교한다.
- 결과가 `KEEP / TUNE / REJECT / TEST_IN_PLAY` 중 하나로 판정된다.
- 변경 수치는 JSON 단일 정본과 런타임·테스트·문서에 함께 반영된다.
- Godot·데이터·운영체계 회귀가 통과한다.
- 사람 체감·Android·장시간 성능 미실행은 `NOT_RUN`으로 남는다.

## 이번 준비 작업의 제외 범위

- 실제 시뮬레이터 구현·실행
- 강화·제작·판매 수치 변경
- 방문 검투사 판매 구현
- 저장·복귀·방치 보상
- Android APK·AAB·실기기 검증
- 사람 시각·접근성·성능·장시간 자동 단조 검증
