# Decision Log

## DEC-001 Android 모바일과 Google Play

- 상태: 확정
- 결정: Android 모바일을 1차 플랫폼으로 하고 Google Play로 출시한다.
- 기술: Godot 4.7.1, GDScript, 세로형 720×1280 기준, AAB 배포, API 36 준비
- 위험: 화면 비율, 발열, 장시간 입력, 저장 복귀, 정책 변경
- 재검토: 첫 Android 기기 Prototype에서 입력 피로 또는 화면 구조 문제가 확인될 때

## DEC-002 제작 입력

- 상태: 확정·구현
- 빠른 연속 터치는 피버를 채우며 불이익을 주지 않는다.
- 제작 마감 정밀작업은 ON/OFF 가능하다.
- 강화 정밀 판정은 +10 단위 특수 강화에서만 사용한다.

## DEC-003 무기 성장

- 상태: 확정·구현
- 기본 제작은 수식어 없는 무기를 만든다.
- 최대 강화 단계는 +100이다.
- +10 단위 특수 강화에서 수식어를 추가·강화한다.
- 고정 공격력 덧셈이 아니라 현재 강화 적용 공격력을 기준으로 점진 성장한다.

## DEC-004 재료와 숨은 무기

- 상태: 확정
- 주재료·보조재료·촉매 세 슬롯을 사용한다.
- 보조재료와 촉매는 +10 단위 특수 강화에서만 표시·적용한다.
- 특정 조합은 숨은 고유 무기를 만든다.
- 숨은 무기 실제 변환은 First Playable 전 별도 범위다.

## DEC-005 판매

- 고객 직접 방문과 상인 납품 의뢰를 사용한다.

## DEC-008 강화 실패·하락·파괴·보정

- 상태: 확정·구현
- +1~+10은 실패 시 기본적으로 단계 유지다.
- +11부터 단계 하락 가능하다.
- +30부터 무기 파괴 가능하다.
- 고단계일수록 하락·파괴 위험과 하락 단계가 증가한다.
- 실패 보정은 실패 횟수에 따라 다음 성공률을 올리고 성공 시 초기화한다.
- 단계 하락 뒤에도 실패 보정을 유지한다.
- 파괴 무기는 보관·수리할 수 없다.
- 이유: 고단계 성공의 가치와 긴장을 키우되 학습 구간은 보호한다.
- 재검토: 실제 +100 플레이테스트에서 손실 피로와 복구 비용을 측정할 때

## DEC-009 수식어 판정

- 상태: 확정·구현
- 특수 강화 보조재료 성질로 수식어 추가 방향을 결정한다.
- 숫돌 → `날카로운`
- 화염석 → `불타는`
- 정령의 심장 → `정령이 깃든`
- +10 첫 수식어 1티어
- +20 첫 수식어 2티어
- +30 두 번째 수식어 1티어
- +40 두 번째 수식어 2티어
- +50 세 번째 수식어 1티어
- +60 세 번째 수식어 2티어
- +70·+80·+90 각 수식어 3티어
- +100 전체 4티어

## DEC-010 강화 분류와 조작 주기

- 상태: 확정·구현
- 일반 강화: +10 단위를 제외한 모든 단계, 원클릭 즉시 판정
- 일반 강화는 보조재료·촉매·정밀 판정을 사용하지 않는다.
- 특수 강화: +10·+20·…·+100
- 특수 강화는 보조재료·촉매 선택 후 정밀 판정을 진행한다.
- 이전 특수 강화 선택 효과는 일반 강화에 적용되지 않는다.
- 이유: 9회의 빠른 강화 뒤 1회의 선택·타이밍 이정표를 배치한다.

## DEC-011 +100 성공률·성장·가격 곡선

- 상태: Prototype·구현
- 10단계 성공률 패턴을 반복하고 높은 구간일수록 기본 성공률을 점진적으로 낮춘다.
- 실제 확률·보정·최저값은 `data/crafting/enhancement_balance.json`이 책임진다.
- 공격력은 현재 강화 적용 공격력 기준으로 증가한다.
- 판매가와 강화 비용은 고단계에서 가속된다.
- 화면에 다음 공격력·판매가·비용·성공/유지/하락/파괴 확률을 표시한다.
- 재검토: 20~30분 플레이와 +100 완주·자동 반복 경제 관찰 후

## DEC-012 단조 방식

- 상태: 확정·구현
- 균형 단조: 기본값
- 안정 단조: 비용 증가, 파괴 방지, 하락 위험 감소
- 폭주 단조: 성공 시 낮은 확률로 총 2단계 상승
- 폭주 단조는 특수 강화와 +9·+19·+29·… 구간에서 사용하지 못한다.
- 이전의 성장·가격·파괴 위험 직접 배율 폭주 구조는 폐기한다.
- 이유: 폭주의 보상을 직관적인 단계 도약으로 집중하고 특수 강화 이정표를 보존한다.

## DEC-013 무기 보관함

- 상태: 확정·구현
- 최대 6개를 보관한다.
- 강화 도중 원하는 단계에서 보관할 수 있다.
- 공격력·판매가·누적 비용·손익·수식어·촉매·마감 품질을 표시한다.
- 파괴 무기는 보관하지 않는다.
- 보관함은 고객·상인 판매의 선택 입력이 된다.

## DEC-014 자동 단조

- 상태: 확정·구현
- 목표 강화 단계·반복 여부·단조 방식·특수 보조재료·촉매를 지정한다.
- 목표 도달 시 자동 보관한다.
- 반복 설정 시 보관함이 찰 때까지 새 철검을 진행한다.
- 지정 재료 재고가 없으면 해당 재료 없이 계속한다.
- 골드 부족·보관함 가득 참·수동 중지에서 종료한다.
- 파괴 시 반복 설정에 따라 새 무기로 재시작하거나 종료한다.
- 이유: +100 반복 피로를 줄이면서 목표·위험·재료 선택은 유지한다.
- 위험: 자동화가 수동 광클·선택의 재미를 무효화하거나 장기 경제를 파괴할 수 있다.
- 재검토: 실제 재화·재료 획득과 저장·복귀를 연결할 때

- 베팅은 판매 루프가 검증된 뒤 별도 PoC에서 경제 악용, 손실 압박, 연령 등급, 보상 역할 중복을 검토한다.

- 재개 조건: MVP-003 플레이테스트와 기본 재화 경제가 안정적으로 동작하고, 베팅이 강화·판매 핵심 재미를 약화시키지 않는다는 가설을 검증할 수 있을 때

## DEC-017 제작 마감 품질의 성능·가치 반영

- 상태: 확정·구현
- 보통 마감은 공격력 ×1.00, 제작 가치 ×1.00이다.
- 좋은 마감은 공격력 ×1.05, 제작 가치 ×1.05이다.
- 완벽한 마감은 공격력 ×1.10, 제작 가치 ×1.12이다.
- 원본 기본 공격력과 품질 적용 기본 공격력을 별도 필드로 보존한다.
- 강화 성장과 판매 가치는 품질 효과를 이어받으며 보관 기록에서도 확인할 수 있다.
- 반복 자동 단조의 새 철검은 보통 마감으로 시작한다.
- 구형 단일 `quality_multiplier` 필드는 사용하지 않는다.
- 이유: 제작 타이밍 선택에 실질 보상을 주되 강화의 복리 성장 때문에 초반 품질 격차가 과도하게 확대되지 않게 한다.
- 재검토: 실제 판매 루프와 20~30분 성장 세션에서 품질별 기대 수익을 측정할 때

## DEC-018 품질별 실제 정수 공격력과 검증 종료코드

- 상태: 확정·구현
- 철검 Prototype 원본 공격력은 20이다.
- 보통·좋음·완벽 마감의 실제 정수 공격력은 각각 20·21·22다.
- 공격력 배율 ×1.00·×1.05·×1.10은 유지하되, 정수 반올림 뒤에도 세 품질이 엄격히 구분되어야 한다.
- 공격력 단위를 10에서 20으로 바꾼 경제 영향은 판매가 공격력 계수 12.0→5.3 보정으로 상쇄한다.
- 테스트 PASS 문구만으로 성공을 판정하지 않고 각 Godot 테스트 프로세스의 종료코드·오류 로그·예상 PASS 표식을 모두 확인한다.
- 성공 분기에서 `quit(0)`을 호출한 테스트 러너는 즉시 `return`해 마지막 실패용 `quit(1)`로 fallthrough하지 않아야 한다.
- 이유: 원본 10에서는 GOOD과 PERFECT가 모두 11로 반올림되어 공격력 보상이 무효화됐고, 기존 정적 검사는 Workflow의 실제 실패 제어 흐름을 확인하지 못했다.
- 재검토: 다른 주재료·무기 기본 공격력을 추가하거나 판매가 계산식을 교체할 때 품질별 정수 구분과 기준 가격을 다시 계산한다.

## DEC-019 Godot AI 벤더 개발 연동

- 상태: 적용·자동 검증 완료 / 로컬 연동 미검증
- `addons/godot_ai/`를 외부 개발 도구 벤더 경계로 유지한다.
- `project.godot`에서 `res://addons/godot_ai/plugin.cfg`를 활성화하고 `_mcp_game_helper`를 오토로드한다.
- 운영 감사는 애드온의 필수 런타임 진입점과 `project.godot` 선언을 검사한다.
- 애드온 소스가 upstream 저장소의 `docs/`·`tests/` 경로를 언급하는 것은 프로젝트 로컬 참조로 판정하지 않는다.
- 이유: 배포용 애드온은 upstream 전체 문서·테스트 트리를 포함하지 않으므로, 벤더 내부 주석·원격 URL 경로와 Blacksmith 책임 원본을 분리해야 한다.
- 로컬 `uv` 설치, MCP 서버 기동, Codex 등 클라이언트 연결은 환경 의존 검증으로 별도 증거가 있을 때만 PASS 처리한다.

## DEC-020 제작 피버 결과 보너스

- 상태: 확정·구현
- 피버 1회 이상이면 공격력 ×1.05·제작 가치 ×1.03을 적용하고 추가 발동은 중첩하지 않는다.
- 마감·피버는 별도 필드로 보존하며 최종 제작 배율은 기준 ×1.00 위 가산 합성한다.
- 피버 적용 보통/좋음/완벽 공격력은 21/22/23, 가치는 ×1.03/×1.08/×1.15다.
- 반복 자동 단조의 새 철검은 보통 마감·피버 미적용이다.
- 이유: 수동 광클 의미는 남기되 제작 지연 반복 적립과 강화 복리 과증폭을 막는다.
- 재검토: 20~30분 플레이의 피버 달성률·입력 피로·수동/자동 효율을 측정할 때

## DEC-021 강화 실패 정책 단일 정본과 의미 검증

- 상태: 확정·구현
- 성공률·실패 보정·단계 하락·파괴·단조 방식 위험 수치는 `data/crafting/enhancement_balance.json`만 책임진다.
- `data/crafting/enhancement_milestones.json`은 +10 단위 수식어 성장과 일반/특수 강화 분류만 책임지며 실패 정책을 포함하지 않는다.
- 선택 재료는 강화 시도 시작 시 소비하고, 세션 시작 자체가 실패하면 거래 계층이 복구한다.
- 성공하면 실패 보정을 초기화하고, 단계 유지·단계 하락에서는 실패 보정을 누적하며, 파괴는 무기 파괴와 세션 종료로 처리한다.
- 최대 +100에서 실제로 접근 가능한 위험 decade는 0~9이며 그 밖의 키는 데이터에 둘 수 없다.
- 성공률 패턴·보정 범위·위험 단조성·이정표 순서·재료 효과·숨은 레시피 참조를 데이터 의미 검증으로 검사한다.
- 기존 성공률·성장·하락·파괴·재료 소비 플레이 수치는 변경하지 않는다.
- 이유: 실행되지 않는 구형 정책과 도달 불가능 데이터를 제거하고 데이터가 실제 런타임 계약을 한 가지 의미로만 설명하게 한다.
- 재검토: 위험·가격 곡선 시뮬레이션에서 수치 자체를 조정할 때 의미 검증 범위와 함께 갱신한다.
