# Blacksmith Game Design Skill

## Skill Modes

- `frame`: 현재 문제와 핵심 경험·제약·결정 질문을 고정한다.
- `update-system`: 승인된 제작·강화·판매 규칙을 책임 원본과 데이터 계약에 반영한다.
- `balance-review`: 성장·성공률·위험·가격·재료 선택의 곡선과 악용 가능성을 검토한다.
- `poc-check`: 가장 위험한 가설, 최소 검증 범위, 성공·중단 기준을 정한다.

## Rules

- 간단한 모바일 광클 경험을 최우선으로 둔다.
- 직원과 수리 시스템을 승인 없이 추가하지 않는다.
- 일반 강화와 +10 단위 특수 강화를 분리한다.
- 보조재료·촉매·정밀 강화는 특수 강화에서만 적용한다.
- 이전 선택이 일반 강화에 몰래 영향을 주지 않게 한다.
- 고단계 보상 확대는 위험·비용·복구 수단과 함께 설계한다.
- 설명 규칙과 실제 JSON 값을 중복 소유하지 않는다.
- 기능 목록이 아니라 플레이어 행동→피드백→보상→다음 선택을 설명한다.
- 외부 벤치마크는 `ADOPT / ADAPT / AVOID / TEST / IGNORE`로 결정에 연결한다.

## Output

- 변경할 책임 원본과 실제 데이터 필드
- 유지·변경·제외 범위
- 정상·실패·경계 플레이 결과
- 관찰 가능한 검증 기준
- 미확정 가설과 다음 PoC

## Validation

- 고객 방문과 상인 납품의 역할이 겹치지 않는가?
- 재료가 특수 강화 수식어와 가치에 실제로만 영향을 주는가?
- 일반 강화에 촉매·정밀 효과가 누출되지 않는가?
- 고단계 효과·가격·위험 증가가 같은 곡선을 반복하지 않고 의미 있는 선택을 만드는가?
- 직접 플레이와 자동 단조가 서로를 무효화하지 않는가?
- 문서 주장과 JSON·GDScript 구현이 일치하는가?

## Failure conditions

- 광클 핵심 경험보다 부가 시스템이 앞선다.
- +10 특수 강화 경계를 건너뛴다.
- 수치의 책임 원본이 Markdown과 JSON에 중복된다.
- 플레이테스트 없이 자극 강화를 확정 규칙으로 승격한다.
